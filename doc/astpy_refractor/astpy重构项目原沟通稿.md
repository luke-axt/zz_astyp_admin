# 整体架构
## 现状
我有一个python项目：astpy。技术栈：python+window+mysql/sqlite,包含如下功能：
1. 实时数据采集：通过rpa、api等方式从外部系统或者公司外购系统采集数据。
2. 离线数仓：以数据处理和指标计算为主，主要就是跑sql。
3. window桌面应用：基于pyside6的桌面应用，主要给用户下载报表，维护业务数据用的。
4. rpa模块：负责跨系统的重复性操作的自动化，或者支持erp无法实现的关键功能（比如现在的尾程，或者外贸业务订单同步等）。
5. 定时任务模块：所有数据采集、离线数仓、rpa等需要定时启动的任务，都由定时任务模块负责调度，主要是通过apschedule实现，目前已经实现的分布式运行、T+1跑批的事件依赖等。
6. 有些需要7*24小时处理的任务，我会独立部署在云服务器，跟我内网生产环境隔离开。（我内网是没有不间断电源的）

整个项目相当于一个桌面应用+大量批处理任务，批处理任务已经超过200个。系统已经对接8个海外仓系统、1个erp、1个金蝶云星空系统，还集成了百度云、腾讯云、163邮件、企业微信等基础服务能力。

目前我们是部署在window服务器，方便做爬虫。另外，有些需要7*24小时的任务，我们是做基于sqlite的单独服务部署在云服务器上，最大程度节约硬件成本。由于我们从0-1的时候架构设计不合理，现在打算重构一下项目。

我们现在不写代码，只讨论设计方案。

## 重构方案
### 一、总体原则
1. 单一入口，统一继承：所有模块（服务、批处理、客户端、RPA、ETL）都从 BaseComponent 继承。
2. 配置驱动： 支持开发/测试/生产环境切换，通过配置文件或环境变量控制。
3. 多数据库支持：可动态切换数据库用户、连接池、方言（SQLite/MySQL/PostgreSQL 等）。
4. 可插拔架构：RPA 等模块可独立打包部署，自带 SQLite 数据库。
5. 日志、监控、错误处理统一：基类封装通用功能，避免重复代码。
6. 我不需要数据库连接池模块，因为我的项目是主要是定时任务，不是在线服务，定时任务都是短平快的任务，处理完就结束。
7. 我不需要api和rpa灵活切换的架构设计，因为项目的200多个作业运行2年多，没有这种场景出现过。
8. 我会保持一个任务一个文件一个类的原则，保持高内聚设计。
9. core库用来沉淀通用逻辑和规则，任务类来处理特定流程的编排，task类负责所有批处理任务的分布式定时调度。

### 一、项目目录结构规划
你可以尝试读取一下D:\app\astpy的一级目录，原来的目录还是比较混乱的。
我现在计划按照如下目录规划项目目录。
用 “核心库（Core）” 来沉淀通用的业务规则，以及项目基础框架。
脚本来放在etl和rpa两个目录下，这里负责数据整理和流程编排。
tasks 负责定时任务调度，包括动态导入脚本类，负责分布式调度和定时任务相关的异常记录和处理。
client 对应来源的pyside6模块，但未来我估计会将这个模块替换到企微机器人就行了。

我目前重构的方向是这样的，帮我看看有哪些不合理的地方：



第一个问题：我想将项目目录拆分成5个一级模块，pyside6桌面、定时任务、数据仓库、rpa、全局系统服务配置管理。这个目录结构是否合理？有哪些优化建议？另外，我还有一个全局都会用到的，utils工具类、外部系统api集成模块，这两个模块应该独立一个一级目录，还是放到哪里去呢？

第二个问题：domain是否必须存在（领域驱动设计）？不需要，因为domain一般适用于基于状态的项目，我们这种项目是批处理项目，不需要domain。
我们能够公用的代码尽量放在core模块。

第三个问题：是否要考虑策略模式，实现rpa和api两种模型灵活切换？不需要。因为两种模式就是两套代码，相当于每个功能要写两遍。且200多个任务几乎没有这种场景存在。

# python项目优化点：
1.将整个python项目改造成从基类继承。
2.清理配置文件，将数据库连接调整到环境变量中，其他配置保留到数据库中，支持mysql和sqlite两种数据库。
3.删除’pyc_db’这个数据库连接配置，（需要改造rptservice，现在rptservice中有优化）
4.删除yml配置文件
5.清理掉无效的旧脚本，尤其是离线批处理脚本。
6.定时任务，一个时间只能运行一个作业id。加id锁
7.记录作业运行时间，每次运行都记录，方便优化。
8.提供停止定时任务的功能，不要直接叉掉cmd窗口。停止任务的时候记录正在运行的jobid到错误日志中，方便重启后人工处理。
9.取消dwd/DwService.py和dwd/DmlPyjob.py两个脚本，所有sql都包装到python脚本中，直接调度python脚本，让逻辑更加灵活，支持运行前、运行后的检查。
10.优化报警机制，超时类别的作业只保存一行报警日志即可。
11.错误日志增加类型，支持error，check等类型，方便统计。
